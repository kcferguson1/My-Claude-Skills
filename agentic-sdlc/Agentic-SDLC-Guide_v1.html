Phase 01 · Ideation & Discovery

Ideation & Discovery

Define the problem worth solving before a single line of code is written. This phase ends with a shared, documented understanding of the problem, the user, and the competitive landscape — and a team decision to proceed.

1
What Happens
  • Problem framing and user research
  • Competitive landscape review
  • Technical feasibility assessment
  • Go / no-go alignment on the opportunity
Agent Integration Points
Problem decomposition Competitor research Assumption mapping Feasibility analysis Opportunity brief drafting
Exit Artifacts
  • Problem statement (1-sentence)
  • Competitive landscape summary
  • Feasibility notes
  • Opportunity brief (1-pager)
Flex point: For pure internal tools or well-understood domains, compress this to a single opportunity brief. Reserve the full competitive analysis for externally-facing products where differentiation matters.
Prompt Library
P1.1 Problem Framing Before any scoping ▾ 
I'm exploring a potential software product idea. Help me rigorously define the problem before I commit to building anything.

Here's my rough idea: [DESCRIBE YOUR IDEA IN 2-3 SENTENCES]

Work through this with me:

1. Restate the problem in user-centric terms — not solution terms. Who experiences this problem? When? What do they currently do instead?

2. Identify the core pain type:
   - Frequency problem (happens constantly, low severity each time)
   - Intensity problem (severe when it occurs, even if rare)
   - Reach problem (affects a very large population)
   Which type is this, and why does that matter for our product strategy?

3. What assumptions am I making? List the top 5 assumptions this idea depends on being true. For each: what's the evidence it's true, and what would falsify it?

4. What would make this problem NOT worth solving? Existing solutions that are "good enough," niche too small, regulatory barriers, technical impossibility — list the real reasons we might walk away.

5. Give me a one-sentence problem statement in this format:
   [User type] struggle to [do X] when [context/trigger], which causes [negative outcome].

Be skeptical. Challenge my framing. I'd rather surface a flawed premise now than build the wrong product.
P1.2 Competitive Landscape Analysis Market research phase ▾ 
I'm building [PRODUCT DESCRIPTION]. Before I finalize scope, help me map the competitive landscape thoroughly.

Context:
- Target user: [WHO THEY ARE AND WHAT THEY DO]
- Core value we're trying to deliver: [1-2 SENTENCES]
- Team size: 2-4 developers
- Initial target market: [GEOGRAPHY / SEGMENT]

Analyze:

1. Direct competitors — products solving the same problem for the same user. For each:
   - Product name and URL
   - Core value proposition
   - Pricing model
   - Key strengths
   - Known weaknesses or gaps users complain about (check G2, Reddit, App Store reviews)

2. Indirect competitors — tools users currently cobble together (spreadsheets, manual processes, Notion, generic SaaS). Why do they settle for these?

3. Emerging threats — any AI-native or recently funded entrants that could commoditize this space within 2 years. Be specific about capability gaps we'd be racing against.

4. Differentiation opportunities — where are the genuine white spaces? What would make our product genuinely defensible rather than a feature-equivalent clone?

5. Build vs. buy signal — are there open-source libraries, APIs, or existing SaaS building blocks that make part of this a solved problem? Should we build around them or compete with them?

6. Positioning recommendation — given all of the above, suggest 1-2 positioning angles that would let a 2-4 person team compete without fighting the incumbents head-on.

Be direct. If there's a well-funded competitor that already does this well, say so explicitly rather than softening it.
P1.3 Technical Feasibility Assessment Before committing to build ▾ 
I want to assess the technical feasibility of this product concept before committing to build it.

Product concept: [DESCRIBE IN DETAIL — core features, user interactions, data flows]
Assumed tech stack: [e.g., React + TypeScript frontend, Node.js API, PostgreSQL, AWS]
Team: 2-4 full-stack developers, comfortable with AI tooling, moderate cloud experience

Evaluate:

1. Complexity tiers — break the product into its core components and rate each:
   Low / Medium / High complexity. Flag any "hard problems" explicitly:
   - Real-time collaboration or sync
   - ML/AI inference in the critical path
   - Payment processing and financial compliance
   - Complex search or recommendation
   - Regulatory requirements (HIPAA, SOC 2, GDPR)
   - High-cardinality data at scale

2. Third-party dependencies — what APIs, platforms, or services does this rely on?
   For each: reliability track record, pricing at scale, lock-in risk, alternatives.

3. Time-to-MVP estimate — given a 2-4 person team working full-time, what's a realistic calendar range for a shippable MVP that could be user-tested? Give a range, not a point estimate, and explain the spread.

4. Top 3 technical risks — what are the specific things most likely to blow up the timeline or force a pivot? For each: likelihood, impact, and mitigation.

5. Proof-of-concept recommendation — what's the smallest technical spike (1-3 days) that would validate the riskiest assumption? What would a passing result look like vs. a result that should make us reconsider?

Be direct. If there is a hard technical blocker or a dependency that makes the timeline unrealistic, say so explicitly. Optimistic feasibility assessments are the most expensive kind.
P1.4 Opportunity Brief Team alignment doc ▾ 
Based on our discovery work, help me write a 1-page opportunity brief to align the team before we move to requirements.

Input context:
- Problem statement: [PASTE YOUR PROBLEM STATEMENT]
- Target user description: [DESCRIBE]
- Competitive landscape summary: [PASTE OR SUMMARIZE KEY FINDINGS]
- Our differentiation angle: [DESCRIBE]
- Feasibility assessment summary: [PASTE KEY POINTS]

Write the brief with these sections:

1. The Problem (2-3 sentences, user-centric — no solution language)

2. Why Now — what has changed in technology, market behavior, or infrastructure that makes this the right moment to build this? (Not just "the problem exists" — why is NOW the window?)

3. Target User — a specific, representative user persona in 3-4 sentences. Name, role, context, and the specific moment this product would change their day.

4. Our Approach — what we're building and why it's different. Not a feature list — a value narrative. What does a user feel or gain that they couldn't before?

5. Success Metrics — how we'll know in 90 days if we're building the right thing. 2-3 specific, measurable signals (not vanity metrics).

6. What We Are NOT Building — explicit scope boundaries. This section is as important as the rest. If it's not here, it's in scope.

Constraints: direct tone, no marketing language, no fluff. This is for internal alignment, not external pitch. 350-450 words max. Our team reads this before the first requirements meeting.
⬡
Quality Gate — Exit Criteria
  • Problem statement written and agreed on by the full team
  • At least one competitor reviewed in depth; differentiation angle defined
  • Feasibility assessment complete; no unresolved hard blockers
  • Opportunity brief reviewed and signed off by all decision-makers
  • Team has explicitly decided: build this, pivot the idea, or stop

Phase 02 · Requirements & Specification

Requirements & Specification

Translate the opportunity brief into a precise, developer-ready specification. Every story should be independently implementable, testable, and scoped to a single behavior. Ambiguity at this stage is a debt that compounds in every phase that follows.

2
What Happens
  • PRD drafted and reviewed
  • User stories written per epic
  • Acceptance criteria written in BDD format
  • NFRs defined and agreed
  • Scope explicitly bounded
Agent Integration Points
PRD drafting Story generation Acceptance criteria NFR checklist Scope prioritization Ambiguity detection
Exit Artifacts
  • Product Requirements Document (PRD)
  • Story map with acceptance criteria
  • Non-functional requirements checklist
  • Explicit out-of-scope list
  • Open questions log (with owners)
Flex point: On very small internal tools, a PRD can be a single GitHub Issue with acceptance criteria. The structure below is for anything customer-facing or with multiple contributors. Match formality to consequence.
Prompt Library
P2.1 Full PRD Generation First requirements pass ▾ 
Write a Product Requirements Document (PRD) for the following product.

Opportunity brief: [PASTE YOUR OPPORTUNITY BRIEF]
Team: 2-4 developers, full-stack, AI-tooling fluent
Platform: SaaS web app, cloud-deployed (AWS/Azure/GCP)
Timeline goal: MVP in [X weeks/months]

Structure the PRD with:

1. Overview
   - Problem (1 paragraph, user-centric)
   - Solution (1 paragraph — what we're building and the core value mechanism)
   - Goals for MVP (3-5 bullet points — measurable outcomes, not features)

2. User Personas (2-3)
   For each: name, role, primary goal, key frustration this product resolves, and one representative scenario.

3. User Stories — organized by epic
   For each story: "As a [user], I want to [action] so that [outcome]."
   Write at minimum 10-15 stories covering the complete core workflow.
   Label each: [MUST / SHOULD / COULD] for MVP prioritization.

4. Acceptance Criteria — BDD format for each story
   Given [initial context] / When [action taken] / Then [expected outcome]
   Include: happy path, empty/zero state, and at least one error case per story.

5. Non-Functional Requirements
   - Performance: page load target, API response time budget
   - Security: auth standard, encryption requirements
   - Accessibility: WCAG 2.1 AA minimum
   - Browser support
   - Uptime SLA

6. Out of Scope — explicit list of things that are deliberately NOT in v1. If it's not here, it's implicitly in scope.

7. Open Questions — table with: question, why it blocks progress, owner, and deadline.

Flag any stories where a product decision is needed before development can start. Use markdown formatting. Be thorough — gaps in this document become rework in development.
P2.2 Story Refinement & Completeness Review After first PRD draft ▾ 
Review these user stories and acceptance criteria for completeness, testability, and hidden risk.

Stories: [PASTE YOUR STORIES AND ACCEPTANCE CRITERIA]

For each story, evaluate:

1. Value clarity — is the "so that [outcome]" specific and meaningful, or is it vague? Rewrite any that are generic ("so that I can use the product" is not a value statement).

2. Testability — are the acceptance criteria specific enough that a developer and a QA engineer would independently write the same test? Rewrite any criteria that use vague language ("the page loads quickly," "the form works correctly").

3. Missing scenarios — for each story, add Given/When/Then scenarios for:
   - Empty state (no data, first-time user)
   - Error state (failed validation, API failure, network error)
   - Permission edge case (unauthorized user, wrong role)
   - Concurrent or race condition case (if applicable)

4. Complexity estimate — T-shirt size (S / M / L / XL) with a one-sentence rationale. Flag anything XL for breakdown into smaller stories.

5. Hidden dependencies — are there stories that cannot be started until another story is complete? Map the dependency chain and flag sequencing constraints.

6. Scope creep risk — which stories have a high likelihood of expanding mid-sprint? Flag them and recommend explicit scope guards in the acceptance criteria.

Output format: revised stories with your changes marked [REVISED] and additions marked [ADDED]. End with a summary table: story ID, complexity, dependencies, risk flag (High/Medium/Low).
P2.3 Non-Functional Requirements Checklist Before architecture begins ▾ 
Generate a comprehensive Non-Functional Requirements (NFR) specification for a SaaS web application.

Product context: [DESCRIBE — what it does, who uses it, what data it handles]
Expected users at launch: [e.g., <100 internal / 500 early access]
Expected users at scale: [e.g., 10,000 / 100,000]
Data sensitivity: [e.g., PII only / financial data / health data / no sensitive data]
Regulatory context: [e.g., GDPR-applicable / US only / HIPAA scope / none]

Cover each NFR category with: Requirement / Rationale / How to Verify

1. Performance
   - Initial page load (LCP target)
   - API response time (p50 and p99)
   - Database query time budget
   - Concurrent user handling target

2. Scalability
   - Expected load profile (read-heavy vs. write-heavy)
   - Horizontal vs. vertical scaling strategy
   - Stateless design requirement (for horizontal scaling)
   - Cache strategy recommendation

3. Security
   - Authentication standard (recommend OIDC/OAuth2 — justify if deviating)
   - Authorization model (RBAC / ABAC)
   - Encryption: in transit (TLS 1.2+) and at rest
   - Secrets management approach (no hardcoded secrets, rotation policy)
   - Dependency vulnerability scanning (tool recommendation)
   - OWASP Top 10 mitigations — list the top 5 most relevant to this product type

4. Reliability
   - Uptime SLA (and what "uptime" means for this product)
   - Error rate budget
   - RTO / RPO (if data loss is a concern)
   - Graceful degradation requirements

5. Observability
   - Structured logging requirements
   - Distributed tracing (if applicable)
   - Key alerting thresholds (error rate, latency, disk, memory)
   - Health check endpoint specification

6. Accessibility
   - WCAG 2.1 AA baseline
   - Specific components most likely to have issues in this product type

7. Developer Experience
   - Local dev setup time target (e.g., <30 minutes for a new engineer)
   - CI pipeline time budget (e.g., <10 minutes for full suite)
   - Documentation coverage expectation
P2.4 MVP Scope Prioritization When backlog is too large ▾ 
My feature list is larger than our MVP timeline. Help me cut ruthlessly.

Full feature list:
[PASTE EVERY PROPOSED FEATURE OR USER STORY]

Constraints:
- Timeline: [X weeks] to a shippable, user-testable MVP
- Team: 2-4 developers
- Goal: something we can put in front of real users to generate validated learning

For each feature, evaluate across four dimensions:
1. Core necessity — can the product be meaningfully user-tested WITHOUT this? (Yes/No — no hedging)
2. Build cost — rough estimate in developer-days
3. Learning value — will this feature generate feedback that shapes future decisions?
4. Deferral risk — does deferring this to v1.1 create technical debt or user confusion?

Then sort into three tiers:

TIER 1 — MVP Core
These features must ship. Without them, the product can't be meaningfully evaluated by users. For each: why it's non-negotiable.

TIER 2 — v1.1 (defer with low risk)
High value, but deferrable. For each: what we lose by deferring, and what to watch for in user feedback that signals we should prioritize it.

TIER 3 — Cut for now
Low signal-to-cost ratio, premature optimization, or "nice to have" that doesn't generate learning. For each: one sentence on why it's cut.

Final output: a Tier 1 feature list with total estimated build days. If Tier 1 still exceeds the timeline, flag it explicitly and suggest what to cut from Tier 1 to make it fit.

Be direct. An honest scope cut is worth more than an optimistic MVP that ships 6 weeks late.
⬡
Quality Gate — Exit Criteria
  • PRD reviewed and signed off by all team members and stakeholders
  • Every MVP story has at least one testable acceptance criterion
  • NFRs defined — no "TBD" on security or performance targets
  • Out-of-scope list explicitly written and agreed
  • All open questions assigned an owner with a resolution deadline
  • No stories with XL complexity remain un-broken-down

Phase 03 · Architecture & Design

Architecture & Design

Define the system before building it. The goal is to make key structural decisions explicit — recorded, reasoned, and reversible where possible. Architecture work done here prevents the most expensive rework a small team can face.

3
What Happens
  • Tech stack finalized
  • System design and component diagram
  • Architecture Decision Records (ADRs)
  • Data model and API contract design
  • Security and threat model review
Agent Integration Points
ADR drafting Architecture review Data model design API contract design Threat modeling Stack comparison
Exit Artifacts
  • System architecture doc + diagram
  • ADRs for key decisions
  • Data model (ER diagram + annotations)
  • API contract spec (OpenAPI or markdown)
  • Threat model notes
Flex point: Write ADRs for decisions that are hard to reverse and have real alternatives. You do not need an ADR for every choice. A wrong ADR is better than no ADR — revisit triggers are part of the format.
Prompt Library
P3.1 Architecture Decision Record (ADR) Any major technical decision ▾ 
Help me write an Architecture Decision Record (ADR) for the following technical decision.

Decision context: [DESCRIBE THE DECISION — e.g., "choosing a primary database," "API layer: REST vs. GraphQL," "auth provider selection," "background job processing approach"]

Product context:
- SaaS web app, 2-4 person team
- GitHub CI/CD, cloud deployment (AWS/Azure/GCP)
- Expected load: [DESCRIBE]
- Data sensitivity: [DESCRIBE]
- Team familiarity: [what the team already knows well]

Options under consideration: [LIST 2-4 OPTIONS]

Write the ADR in this format:

---
ADR-[NNN]: [Short title]
Status: Proposed
Date: [TODAY]

Context
[What is the situation forcing this decision? What constraints, forces, or requirements are at play? What would happen if we made no decision and deferred?]

Decision Drivers
[4-6 criteria we're optimizing for, ranked by importance. Be specific — not "scalability" but "must support 10,000 concurrent connections with a 2-person ops team."]

Options Considered
For each option:
- Name
- Description (1 paragraph)
- Pros (specific to our context and decision drivers)
- Cons (honest — include cost, complexity, and operational burden)
- Rough implementation effort estimate

Decision
[Which option, and the primary reason. 2-3 sentences max.]

Rationale
[Explain the tradeoffs accepted and why the chosen option wins on the criteria that matter most for our stage.]

Consequences
[What gets easier? What gets harder? What are we explicitly accepting as a tradeoff?]

Revisit Trigger
[Under what specific circumstances — load threshold, team growth, feature requirements — should we revisit this decision?]
---
P3.2 System Architecture Review After first architecture draft ▾ 
Review this system architecture for a SaaS web application. I want to find gaps before we start building.

Architecture description:
[DESCRIBE YOUR SYSTEM IN DETAIL:
- Frontend: framework, hosting, CDN approach
- API layer: framework, auth method, request routing
- Backend services: what they do, how they communicate
- Database(s): type, hosting, backup approach
- Background jobs: queue, worker setup
- File storage: approach and access pattern
- External integrations: third-party APIs or services
- Deployment: cloud provider, containers or serverless, environments]

Team: 2-4 developers, full-stack

Review against these dimensions — skip anything that is genuinely low-risk for our stage:

1. Single points of failure — what happens when component X is unavailable? Which components lack failover or graceful degradation?

2. Data consistency — where could we have race conditions, stale reads, or write conflicts? Identify any operations that need to be atomic but currently aren't.

3. Security perimeter — trace the path of a single user request. Where does untrusted input enter? Is it validated at every boundary? Are there any implicit trust assumptions between internal services?

4. Scalability bottlenecks — which components will degrade first under load? At what order-of-magnitude of users? Is the cost profile linear or does something get expensive suddenly?

5. Operational complexity — what does an incident look like at 2am? Can we tell what's wrong from logs and metrics alone? What manual interventions are required for common operations (deploy, rollback, schema migration)?

6. Developer experience — can a new engineer run this locally in under 30 minutes? Are there any environment-specific secrets that block local development?

For each issue: describe the risk, severity (Critical / High / Medium / Low), and recommended mitigation. Focus on issues that would be significantly harder to fix after the codebase grows.
P3.3 Data Model Design Before any schema or migration work ▾ 
Design a data model for the following application. I'll implement it in PostgreSQL.

Product description: [DESCRIBE]
Core entities I've identified: [LIST THEM]
Key user workflows that touch data: [DESCRIBE THE 3-5 MOST IMPORTANT DATA OPERATIONS]
Expected query patterns: [e.g., "mostly read-heavy," "users query their own data by date range," "we need full-text search on X"]

Produce:

1. Entity definitions — for each entity:
   - Name and purpose
   - Fields: name, data type (PostgreSQL types), constraints (NOT NULL, UNIQUE, default values)
   - Primary key approach (UUID vs. serial integer — recommend and justify)
   - Relationships to other entities (FK references)

2. Relationship summary — describe the complete ER model in plain text:
   "User has many Projects, Project belongs to exactly one User, Project has many Tasks..."

3. Indexing recommendations — which fields to index and why. Include composite index recommendations for common query patterns. Flag any fields that look like they'll be used in WHERE clauses that lack indexes.

4. Multi-tenancy design — how does this model isolate data between customers?
   Recommend: row-level (tenant_id column), schema-level, or database-level — with reasoning for our scale and team size.

5. Soft delete vs. hard delete — recommendation per entity. Consider: audit requirements, referential integrity, recovery needs.

6. Audit fields — which entities need created_at, updated_at, created_by, updated_by? Are there entities where we need a full audit log (separate events/history table)?

7. Migration risk assessment — which entities are most likely to change as we learn from users? Flag the top 3 and describe what kinds of changes to expect so we can design for extensibility now.

8. Potential schema pitfalls — based on the design, flag anything that looks like it will require a painful migration after data accumulates (e.g., adding a NOT NULL column, splitting a table, changing a FK cascade behavior).
P3.4 REST API Contract Design Before API implementation begins ▾ 
Design the REST API contract for the following feature set.

Features to support: [DESCRIBE THE USER STORIES OR FEATURES THAT REQUIRE API ENDPOINTS]
Auth model: [e.g., JWT Bearer tokens via OIDC, API key for service-to-service]
API versioning strategy: [e.g., /api/v1/ prefix]
Response envelope: [e.g., consistent { data, error, meta } wrapper or flat responses]

For each endpoint, define:
- HTTP method + path
- Description (what this endpoint does and when to call it)
- Path parameters (name, type, description)
- Query parameters (name, type, required/optional, default, description)
- Request body schema (JSON — field name, type, required/optional, validation constraints, example)
- Success response (status code + body schema with example)
- Error responses (status code + error schema for each failure case — not just 400/500 generics)
- Auth requirement (authenticated / public / specific role/scope)
- Idempotency (is this endpoint idempotent? if not, what makes it safe to retry?)

After defining all endpoints, also address:

1. Pagination — which endpoints return lists? Specify the pagination strategy:
   - Cursor-based (recommended for real-time data) or offset-based (simpler but inconsistent under inserts)?
   - Request params and response envelope for pagination metadata

2. Standard error envelope — define a single error response schema to use consistently:
   { error: { code, message, details?, requestId? } }

3. Rate limiting headers — which endpoints need rate limiting? Specify the X-RateLimit headers to return.

4. Async operations — any endpoints that trigger long-running work? Specify whether to use polling (return a job ID), webhooks, or SSE.

5. Backwards compatibility rules — what's our policy for breaking vs. non-breaking API changes? What triggers a version bump to /v2/?

Output as structured markdown. Include a one-line curl example for each non-trivial endpoint.
⬡
Quality Gate — Exit Criteria
  • ADRs written for: database, API layer, auth approach, and any other hard-to-reverse decision
  • System architecture reviewed by at least two team members; no Critical-severity issues unresolved
  • Data model reviewed; all entities have defined indexing and multi-tenancy strategy
  • API contract defined for all MVP endpoints; error envelope standardized
  • Threat model reviewed; OWASP Top 10 gaps addressed or explicitly accepted

Development

The agent is your pair programmer, reviewer, and documentation writer — not your replacement. Every AI-generated code block gets the same human review as human-written code. The branching strategy is the communication protocol of the team.

4
What Happens

  • Code reviewed on every PR
  • Documentation written alongside code
  • Agent-assisted debugging for blockers
Agent Integration Points
Implementation planning Code generation Code review Debugging Refactoring Inline documentation Commit message writing
Exit Artifacts
  • Merged, reviewed feature branches

Flex point: On a 2-person team, a full async PR review cycle may be overkill for small changes. Establish a size threshold — e.g., PRs under 50 lines with passing CI can merge with async approval; anything larger requires synchronous review. Write this rule down and apply it consistently.

P4.1 Feature Implementation Plan Before writing any code for a story ▾ 
I'm implementing this user story. Before I write any code, help me plan the implementation.

User story: [PASTE THE STORY]
Acceptance criteria: [PASTE ALL GIVEN/WHEN/THEN CRITERIA]

Tech stack:
- Frontend: [e.g., React + TypeScript, React Query for data fetching, Tailwind for styles]
- API: [e.g., Node.js + Express + Zod for validation]
- Database: [e.g., PostgreSQL via Prisma ORM]
- Auth: [e.g., JWT via Auth0 / Supabase Auth]
- Deployment: [e.g., AWS ECS via GitHub Actions]

Existing patterns to follow: [DESCRIBE ANY CONVENTIONS — e.g., "repository pattern for DB access," "all mutations go through service layer," "React components use compound component pattern," "error handling uses a Result type"]

Produce:

1. Implementation steps — ordered list from data layer to API to UI. Be specific about what changes at each layer and in what sequence. Include "write tests" as explicit steps, not an afterthought.

2. Files to create or modify — for every file: path, what changes, why. Group by layer (DB migration → model/repository → service → API route → API test → frontend component → component test).

3. Database changes needed — new tables, columns, indexes, or migrations. Write the migration in SQL or Prisma schema syntax.

4. API changes needed — new or modified endpoints. Specify method, path, request/response schema changes.

5. Edge cases to handle — list every case that must not slip through:
   - Permission check (who can do this? what happens if they can't?)
   - Empty / null / zero state
   - Duplicate or idempotency case
   - Failure of a downstream dependency

6. What NOT to do — common over-engineering traps or out-of-scope additions to avoid.

Do not write implementation code yet. I want to review this plan first.
P4.2 Substantive Code Review On every PR — before human review ▾ 
Review this code for correctness, security, performance, and maintainability. Do not comment on style or formatting — that's handled by our linter.

Code diff / files:
[PASTE CODE OR DESCRIBE THE PR]

Context:
- What this code does: [DESCRIBE]
- Which user story it implements: [PASTE STORY + ACCEPTANCE CRITERIA]
- Stack: [TypeScript / Node.js / React / PostgreSQL / etc.]
- Where this lives in the system: [e.g., API route handler, React component, database service layer]

Review for:

1. Correctness
   - Does it fulfill all the acceptance criteria?
   - Are there logic errors, off-by-one errors, or incorrect conditional branches?
   - Are there cases in the acceptance criteria that this code doesn't handle?

2. Security
   - SQL injection, XSS, or injection risks (even with parameterized queries, look for dynamic query construction)
   - Authentication bypass — is auth checked before the operation, or after data access?
   - Authorization — is the current user's permission to access THIS specific resource verified (not just "are they logged in")?
   - Sensitive data in logs, error messages, or API responses
   - Secrets or credentials in code or config files

3. Error handling
   - Are errors caught at the right level?
   - Do uncaught promise rejections exist?
   - Do errors expose internal stack traces to API consumers?
   - Is every error logged with enough context to diagnose in production?

4. Performance
   - N+1 query patterns (loop containing a DB call)
   - Unbounded queries (no LIMIT on list queries)
   - Missing async/await causing synchronous blocking
   - Unnecessary data fetching (SELECT * where only 2 columns are used)

5. Testability
   - What's difficult to unit test in the current structure?
   - Are side effects (DB writes, API calls, emails) injectable/mockable?

Format: For each issue — file:line, severity (Critical/High/Medium/Low), description of the problem, and the specific fix or refactoring needed.
P4.3 Systematic Debugging When stuck on a bug for >30 minutes ▾ 
I have a bug I can't track down. Help me diagnose it systematically.

Symptom: [DESCRIBE EXACTLY WHAT IS HAPPENING — include the exact error message, unexpected output, or behavior]

Reproduction steps:
1. [STEP 1]
2. [STEP 2]
3. [WHAT HAPPENS VS. WHAT SHOULD HAPPEN]

Is this reproducible consistently, or intermittently? [CONSISTENT / INTERMITTENT — if intermittent, describe the pattern]

Stack: [FULL TECH STACK]
Environment where it occurs: [local / staging / production / all]

Relevant code:
[PASTE THE CODE MOST LIKELY INVOLVED — the handler, function, or component]

Logs / error output:
[PASTE THE FULL ERROR, STACK TRACE, OR LOG LINES — do not summarize, paste the raw output]

Recent changes: [Any code changes, deploys, dependency updates, or config changes in the last 24-48 hours]

Approach this systematically:
